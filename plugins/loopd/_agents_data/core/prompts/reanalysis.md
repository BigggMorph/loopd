### 판단 기준
피드백과 파이프라인 컨텍스트를 함께 해석하고 다음 중 하나로 판단하세요:

1. **요구사항 변경/확장** → key_requirements를 전체 범위로 업데이트, level/next_agent 재평가
2. **단순 격려/확인 ("좋아", "ㅇㅋ")** → 기존 analysis 유지, 동일한 level/next_agent 반환
3. **부분 수정 + 진행 ("ㅇㅋ 근데 X도 해줘")** → X를 key_requirements에 추가, 나머지 유지
4. **파이프라인 기술적 실패** → backward transition 분류가 `파이프라인_검증_실패`이면 level을 하향하지 말 것. 파이프라인 내부 검증 오류(아티팩트 누락 등)는 태스크 난이도 변화가 아님

### 규칙
- 사용자 피드백이 이전 분석보다 **우선**
- key_requirements에는 피드백 반영 후 **전체 범위**를 포함 (delta만 X)
- reasoning에 "피드백을 어떻게 해석했는지" 반드시 설명
- backward transition 분류가 `파이프라인_검증_실패`인 경우 level 하향 **금지**
